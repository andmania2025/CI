# コードレビュープロンプト

## レビュー重点領域

### 1. コード品質

- コードは `.cursor/rules/` の既存ルールに従っているか？
- コードは読みやすく、構造化されているか？
- コードスメルやアンチパターンはないか？
- エラーハンドリングは適切に実装されているか？

### 2. セキュリティ

- セキュリティ脆弱性はないか？
- ユーザー入力は適切にバリデーションおよびサニタイズされているか？
- 認証と認可は正しく実装されているか？
- 機密データは適切に保護されているか？

### 3. パフォーマンス

- 明らかなパフォーマンスボトルネックはないか？
- Reactフックと状態管理を効率的に使用しているか？
- データベースクエリは最適化されているか？
- 必要な箇所で適切なキャッシングが実装されているか？

### 4. テスト

- コードはテスト可能か？
- 機能をカバーする十分なテストがあるか？
- テストはテストガイドラインに従っているか？

### 5. アーキテクチャ準拠

- コードはアーキテクチャパターンに従っているか？
- コンポーネント構造は適切か？
- 依存関係は適切に管理されているか？
- 関心の分離が維持されているか？

## レビュー手順

1. **建設的に**: 具体的で実行可能なフィードバックを提供する
2. **標準を参照**: 関連するルールやドキュメントを指し示す
3. **改善を提案**: 問題を特定するだけでなく、解決策を提案する
4. **コンテキストを考慮**: 広範な機能とユーザーへの影響を理解する
5. **問題に優先順位をつける**: 重大な修正と「あったら良い」を区別する

## レビューチェックリスト

- [ ] コードはTypeScriptのベストプラクティスに従っている
- [ ] コンポーネントは適切な命名規則を使用している（PascalCase）
- [ ] Server Actionsには適切なエラーハンドリングが含まれている
- [ ] データベースクエリはPrismaを適切に使用している
- [ ] 認証は適切にバリデーションされている
- [ ] スタイリングはTailwindクラスを一貫して使用している
- [ ] テストが存在し、意味がある
- [ ] ハードコードされた値やマジックナンバーがない
- [ ] Propsインターフェースが適切に定義されている
- [ ] アクセシビリティへの配慮がなされている
